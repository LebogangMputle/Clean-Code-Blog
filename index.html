Blog: Clean Code Introduction

Introduction:

In the quest for clean code, mastering the balance between objects and data structures is paramount, as they form the backbone of software architecture, defining the elegance and maintainability of the codebase.

Organization:

Data structures provide a well-defined way to store and organize information. This makes your code clear and easier to understand, both for you and other developers.

Efficiency:

Choosing the right data structure for the task improves code efficiency. For instance, using a hash table for quick lookups is better than iterating through a list every time.

Maintainability:

Clean code with well-defined data structures is easier to maintain and modify. If the data organization is clear, changes can be made without unintended consequences.

Encapsulation:

Objects encapsulate data and related functionalities. This promotes modularity and reduces the risk of errors spreading throughout your codebase.

The right object structures allow for easier code extension. You can add new functionalities without drastically changing existing code.

Blog:Difference between objects and data structures

Objects and Data structres are:

  • Objects hide their data behind abstractions and expose functions that operate on that data. Data structures expose their data and have no meaningful functions.
  • Objects expose behaviour and hide data. This makes it easy to add new kinds of objects without changing existing behaviours. It also makes it hard to add new behaviours to existing objects..
  • Data structures expose data and have no significant behaviour. This makes it easy to add new behaviours to existing data structures (how data can be stored in different structures) but makes it hard to add new data structures to existing functions.

    • The Law of Demeter

  • This law says a module should not know about the inwards (internal workings of a device or machine) of the objects it manipulates. This means that an object should not expose its internal structure through accessors.

    • Hybrids

  • They are half objects and half data structures. Hybrids make it hard to add new functions but also make it hard to add new data structures.

    • Data Transfer Objects

  • The most perfect example of a data structure is a class with public variables and no functions which is a DTO.
  • Very useful structures, especially when communicating with databases or parsing (the process of analyzing a string of symbols) messages from sockets

    • Active Records

  • They are data structures with public (or accessed) variables. They typically have navigated methods.
